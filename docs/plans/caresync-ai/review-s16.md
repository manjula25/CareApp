### Spec axis

**Real defect #1 — found by Commit 2 sub-agent mid-flight, fixed via Option A + pivot decision:**

> `implementation-plan-s16.md` §Commit 2 Phase C1 specified "temperature: 0 + seed: 42 on all 4 agents' `client.responses.create(...)` calls." The Commit 2 sub-agent (`a3374fa2ed26872e4`) executed and surfaced that the OpenAI Responses API **rejects both parameters**: `seed` returns `400 Unknown parameter: 'seed'.` on all models; `temperature` returns `400 Unsupported parameter: 'temperature' is not supported with this model.` on the reasoning tier (`gpt-5.5`). The pin-based variance-collapse strategy was therefore unviable on the current API surface.

Fixed: user picked **"Option A + pivot commit 3"** mid-execution. The pin code was reverted, `apps/api/src/eval/varianceProbe.ts` was written as an observability-only tool, the probe ran successfully (81.25% per-patient agreement on the 16 dev-labeled patients × 3 runs), and commit 3 proceeded unchanged (v2 rubric is independent of substrate stability). The API-constraint finding is documented in `variance-probe.md §"API constraint"` and preserved in `~/.claude/projects/.../memory/openai-responses-api-no-seed.md` as a project-wide invariant ("Do NOT propose either param in future slices"). The pivot decision is also reflected in `verification-s16.md §7` row #1 + #3 (both signals DEFERRED per the API constraint).

**Documented design tradeoff — not a defect:**

> The held-out sensitivity is `null (denominator 0)` rather than a number ≥ the 50% target. This is a property of the held-out labels (`pop-0011..pop-0020`) and `labelFromBundle`'s `riskScoreFor() ≥ 75` threshold, not a v2 rubric failure. Per the rubric-eval-result.md §"Why held-out sensitivity is undefined" section: 8 of 10 held-out patients have 1–2 conditions (`riskScoreFor(2, 24) = 66`, below 75); `pop-0014` has 3 conditions but `recencyHours > 720` (`riskScoreFor(3, >720) = 72`, below 75); `pop-0013` has 2 conditions at the lowest recency bucket (`riskScoreFor(2, ≤72) = 66`, below 75). With `riskScoreFor ≥ 75` as the positive-label threshold, the cohort has 0 positive labels; sensitivity = 0/0 = undefined. Surfaced in verification-s16.md §1 + §5 + §7 matrix row #5; two options for making held-out sensitivity measurable noted (§9 #2): (a) lower the threshold to `riskScoreFor ≥ 65` (most patients still not-high-risk), or (b) extend the procedural generator's condition mix to include more 3-condition patients in the held-out range. Both are label-set changes, out of S16 scope.
