## B. Built vs. Prototyped vs. Envisioned

**Built:** Full FHIR R4 read/write layer against HAPI v7.2.0; SMART Backend Services (assertion minting, exchange, caching, Bearer attachment, **now enforced by HAPI via `docker-compose.yml` config + curl 401/200 evidence**); CDS Hooks discovery + patient-view service; four LLM agents (Risk, Care Gap, SDOH, Action Planner) on `gpt-5.5` with structured function-tool output and streaming; citation enforcement (validation + redaction); per-finding `confidence: 0-1` via `confidenceScorer.ts` heuristic (`apps/api/src/agents/confidenceScorer.ts`, propagated into Action Planner tasks); FHIR Task lifecycle (create, assign, transition, list, detail); FHIR Subscription (rest-hook creation, webhook receiver, SSE relay to web); role-based auth (3 roles × 3 domains) with audit trail; mobile coordinator app (TaskQueue, TaskDetail with complete/defer/escalate); PatientDetail with live Canvas agent graph + SSE streaming; Population dashboard (500-patient cohort, risk×urgency scatter, critical-zone count, **cost-avoidance tile with named-constant formula**); Governance dashboard (audit trail, confidence distribution, demographic parity); Quality/HEDIS measure (diabetes HbA1c testing rate); SDOH resource directory + audited FHIR ServiceRequest referrals; team performance view; 12 Playwright E2E specs; **evaluation harness** (`npm run eval`) with pure `computeMetrics` (sensitivity/specificity/PPV) + `computeErrorAnalysis` modules, 16-patient dev-labeled ground-truth file, cache-first/live-fallback harness, JSON + Markdown report generation, unit tests for both pure modules — **run against live HAPI + LLM** producing committed `docs/eval-report.json` with actual metrics (Care Gap sensitivity 100%/specificity 0%/PPV 90.9%, Risk sensitivity 100%/specificity 30.8%/PPV 25%, SDOH agreement 66.7%) and per-patient error analysis (10 disagreements: 1 Care Gap FP, 9 Risk FPs); **clinician review tool** with bidirectional round-trip — `review:render` generates a self-contained HTML form (localStorage drafts, JSON download), `review:apply` (`apps/api/src/scripts/apply-clinician-review.ts`, 297 lines + 3 unit tests) reads the JSON and writes back into `labels.json` with `source: "clinician"` flips and `clinicianOverride` slot population; S14 SDOH rebalance to 3 positive + 2 explicit-negative.

**Prototyped:** Confidence is emitted on every finding but the per-agent confidence-bucketed accuracy sub-tables in `eval-report.md` are deferred to a follow-up commit (E1 acceptance #4 — plumbing `confidence: number` through `PatientFindings` so `computeMetrics` can group findings is a non-trivial refactor). The HAPI-side SMART config trusts any token signed by the configured public key — correct for the POC's `client_credentials` flow but not the right shape for production multi-actor SMART (documented honestly in `verification-s14.md`).

**Envisioned:** Multi-EHR deployment; population-level outcome tracking (historical snapshots for trend analysis); batch analysis endpoint; formal governance frame alignment (model card, named regulatory pathway, e.g. NIST AI RMF or WHO guidance); **clinician validation of eval labels** (tool is now complete end-to-end — render, apply, regression-tested — but 0 of 16 clinicians have used it); multilingual support; low-connectivity / offline operation; Risk agent calibration to reduce the 9-FP rate; held-out eval set (current 16 are dev-labeled, not held out).

## E. Band, Strongest Dimension, Biggest Risk/Gap

**Band:** **Finalist (85–100).**

**Strongest dimension:** **P1 + P3** together — seven load-bearing HL7 standards feeding a genuinely inventive multi-agent AI architecture with citation enforcement. The standards are the grounding layer that makes the AI safe; the AI is the reasoning engine no rule system could replicate. The two reinforce: the citation validator is meaningful because every FHIR ResourceType/id is a real anchor; the FHIR portability is meaningful because the agents produce structured FHIR Tasks, not proprietary blobs.

**Biggest risk/gap:** **P2 / P4 / P6** — these three pillars share the same constraint: 0/16 clinician-validated labels, the Risk agent's 9-FP rate (specificity 30.8%, PPV 25%), and the absence of a held-out eval set. The infrastructure to close the clinician-validation gap is now **complete** (`review:render` + `review:apply` + unit tests + data-driven disclosure banner), so the remaining work is engagement, not code. Closing any one of the three would lift at least one pillar by 1, and lift the weighted total.

## F. Open Questions for the Team

1. **P2 / P4 (highest priority):** The Risk agent has 9 false positives (specificity 30.8%, PPV 25%) — it over-calls risk on patients with seed `riskScore < 75`. The S13 risk-rubric attempt was reverted after it made things worse (specificity regressed 30.8% → 0% on the held-out 16). Is there a plan to either (a) calibrate the prompt against the 9-FP list, or (b) gate the critical-risk display behind a higher bar so coordinators aren't alerted to medium-risk patients as "critical"?
2. **P2 / P4 / P6:** The clinician review tool is now **complete** (render + apply, regression-tested) but 0/16 labels are clinician-validated. Has the HTML form been sent to any clinician? If not, what is the timeline? This is the single highest-leverage unblock.
3. **P4:** No model card, no NIST AI RMF alignment, no WHO guidance alignment, no named regulatory pathway is evidenced. A single page naming the target governance frame + the audit signals the system already emits (audit trail, demographic parity, citation enforcement, per-finding confidence) would be cheap and meaningful.
4. **P6:** The 16 dev-labeled rows are not a held-out set — they are the ground truth the eval is computed against. The brief's calibration is "would score 5 with a held-out eval set showing sensitivity/specificity." Is there a plan to add either clinician-reviewed labels or a procedurally-generated held-out cohort?
5. **P6:** The most recent eval run covered only 3/16 patients (OpenAI quota exhausted mid-run on 13/16). Is this a one-off or a recurring constraint? If recurring, the eval can only be run on the 3 cached patients, which is not a real eval.
6. **P6:** Per-agent confidence-bucketed accuracy sub-tables (E1 acceptance #4) are deferred to a follow-up commit. What is the timeline? This would close the last remaining P4 holdback about confidence distribution.
7. **P9:** No multilingual support or low-connectivity / offline consideration is evidenced. Is there a plan for non-English populations or graceful-degradation operation for resource-limited settings?
8. **P1 (smaller):** S14 `5e73c68` added HAPI config that trusts any token signed by the configured public key. `verification-s14.md` documents this honestly as POC-correct but not production-shaped. Is there a plan for production multi-actor SMART scope enforcement (per-actor scopes, not just per-server)?
