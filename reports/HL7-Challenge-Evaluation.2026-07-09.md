## B. Built vs. Prototyped vs. Envisioned

**Built:** Full FHIR R4 read/write layer against HAPI v7.2.0; SMART Backend Services (assertion minting, token exchange, caching, Bearer attachment, HAPI-side JWT validation via `docker-compose.yml` config, app-tier enforcement middleware); CDS Hooks discovery + patient-view service; four LLM agents on `gpt-5.5` with structured function-tool output and streaming; citation enforcement (validation + real-time narration redaction); per-finding `confidence: 0-1` via deterministic bundle-evidence heuristic (`confidenceScorer.ts`); FHIR Task lifecycle (create, assign, transition, list, detail); FHIR Subscription (rest-hook creation, webhook receiver, SSE relay to web); role-based auth (3 roles × 3 domains) with audit trail; mobile coordinator app (TaskQueue, TaskDetail with complete/defer/escalate); PatientDetail with live Canvas agent graph + SSE streaming; Population dashboard (500-patient cohort, risk×urgency scatter, critical-zone count, cost-avoidance tile with named-constant formula); Governance dashboard (audit trail, confidence distribution, demographic parity); Quality/HEDIS measure (diabetes HbA1c testing rate); SDOH resource directory + audited FHIR ServiceRequest referrals; 14 Playwright E2E specs; evaluation harness (`npm run eval`) with pure `computeMetrics` + `computeErrorAnalysis` + `labelFromBundle` + `outreachSchema` modules, 26-patient label file (16 dev-labeled + 10 held-out), cache-first/live-fallback harness, JSON + Markdown report generation, unit tests for all four pure modules; clinician review tool with bidirectional round-trip (`review:render` generates self-contained HTML form, `review:apply` applies JSON back to `labels.json`); outreach tracking infrastructure (`clinician-outreach.json`).

**Prototyped:** Per-agent confidence-bucketed accuracy sub-tables in `eval-report.md` deferred (plumbing `confidence: number` through `PatientFindings` for `computeMetrics` grouping is a non-trivial refactor). Held-out SDOH sub-metric reports 0 data points by design (held-out bundles have no AHC-HRSN observations per `population.ts:buildSdohForIndex`). HAPI-side SMART config trusts any token signed by the configured public key — correct for POC's `client_credentials` flow but not production-shaped for multi-actor SMART (documented honestly in `smartAuth.ts:6-19`).

**Envisioned:** Multi-EHR deployment; population-level outcome tracking (historical snapshots); batch analysis endpoint; formal governance frame (model card, NIST AI RMF, named regulatory pathway); clinician validation of eval labels (tool is complete end-to-end but 0 of 26 clinicians have used it); multilingual support; low-connectivity / offline operation; Risk agent calibration to reduce the FP rate.

## E. Band, Strongest Dimension, Biggest Risk/Gap

**Band:** **Finalist (85–100).**

**Strongest dimension:** **P1 + P3** together — seven load-bearing HL7 standards feeding a genuinely inventive multi-agent AI architecture with citation enforcement. The standards are the grounding layer that makes the AI safe; the AI is the reasoning engine no rule system could replicate. The two reinforce: the citation validator is meaningful because every FHIR `ResourceType/id` is a real anchor; the FHIR portability is meaningful because the agents produce structured FHIR Tasks, not proprietary blobs.

**Biggest risk/gap:** **P2 + P4 + P6** — these three pillars share the same constraint: 0/26 clinician-validated labels, the Risk agent's FP rate (2 FPs in the latest run, specificity 0% on n=2), and the fact that the current committed eval report only scored 2/26 patients (14 failed with `--no-live` flag, 0 live runs) with the held-out section explicitly skipped (`--dev-only`). The eval infrastructure is comprehensive — harness, held-out labels, clinician review tool, outreach tracking — but the current committed artifact doesn't demonstrate it working at scale. The infrastructure to close the clinician-validation gap is complete (`review:render` + `review:apply` + tests + data-driven disclosure), so the remaining work is engagement + re-running the eval with live agents, not code.

The held-out infrastructure exists (labels in `labels.json`, `labelFromBundle.ts` with 10 TDD tests). When will the held-out eval be run and its results committed? This is the single highest-leverage unblock for P6 → 5.

3. **P2 / P4:** The Risk agent has 2 false positives in the latest run (specificity 0%, PPV 0% on n=2). Previous runs showed 9 FPs (specificity 30.8%, PPV 25% on n=16). Is there a plan to calibrate the Risk prompt against the FP list, or to gate the critical-risk display behind a higher bar?

4. **P2 / P4 / P6:** The clinician review tool is complete (render + apply, regression-tested) but 0/26 labels are clinician-validated. Has the HTML form been sent to any clinician? If not, what is the timeline? The outreach log ships empty (`data/eval/clinician-outreach.json` with `invitations: []`).

5. **P4:** No model card, no NIST AI RMF alignment, no WHO guidance alignment, no named regulatory pathway is evidenced. A single page naming the target governance frame + the audit signals the system already emits (audit trail, demographic parity, citation enforcement, per-finding confidence) would be cheap and meaningful.

6. **P6:** Per-agent confidence-bucketed accuracy sub-tables are deferred. What is the timeline? This would close the last remaining P4 holdback about confidence distribution.

7. **P9:** No multilingual support or low-connectivity / offline consideration is evidenced. Is there a plan for non-English populations or graceful-degradation operation for resource-limited settings?

8. **P1 (smaller):** The HAPI SMART config trusts any token signed by the configured public key (`docker-compose.yml:25-26`). `smartAuth.ts:6-19` documents this as POC-correct but not production-shaped. Is there a plan for production multi-actor SMART scope enforcement (per-actor scopes, not just per-server)?
